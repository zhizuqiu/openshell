# Cubectl 功能需求文档 (Functional Requirements Document)

本文档旨在汇总并规范 `cubectl` 项目的所有功能需求、用户体验细节及技术约束。

## 1. 项目概述

`cubectl` 是一个 AI 驱动的 Kubernetes 助手，通过交互式命令行界面 (TUI) 帮助用户更高效地管理和查询 Kubernetes 集群资源。

## 2. 核心功能需求

### 2.1 AI 交互引擎 (AI Interative Engine)

- **Agent 化操作**: 系统通过集成的 AI Agent 接收自然语言指令并转换为 Kubernetes API 操作。
- **实时流式反馈**: 采用 LangGraph 的 `streamMode: 'updates'`，确保用户能实时看到 AI 的思考过程和操作进度。
- **上下文感知**: 自动识别当前的 Kubernetes 上下文 (Context) 和命名空间 (Namespace)，并在对话中保持状态。

### 2.2 人类在环决策 (Human-in-the-Loop, HITL)

- **敏感操作拦截**: 对于 `patch`、`create` 等可能修改集群状态的操作，AI 必须暂停并请求用户确认。
- **交互式选择菜单**: 使用 `ink-select-input` 提供 "Approve" (批准) 和 "Reject" (拒绝) 选项。
- **底部菜单设计**: 为避免焦点混淆，决策菜单应呈现在终端底部的输入区域，替换常规的命令提示符。
- **无缝流续传**: 用户确认决策后，AI 必须能在同一个消息块内继续输出后续内容，而不是开启新消息块。
- **批量工具支持**: 在单次 AI 响应包含多个工具调用时，支持集中或顺序的决策逻辑。

### 2.3 自主执行模式 (Autonomous Execution)

- **快捷键开启**: 通过 `Ctrl+A` 全局快捷键快速切换。
- **状态指示器**: UI 显著位置展示当前是否处于 "自主执行" 状态 (✓/✗)。
- **逻辑分支**: 开启时跳过 HITL 环节自动执行；关闭时强制触发人工审核。

### 2.4 消息去重与一致性 (Message Deduplication)

- **多层过滤机制**: 使用 `seenMessageIdsRef` (Set) 追踪已处理的消息 ID，防止 LangGraph 在 `updates` 模式下重复推送中间状态或 middleware 产生的历史副本。
- **即时拦截**: 在流处理循环内部进行 ID 校验，确保界面渲染永远只包含唯一的、增量的操作反馈。

## 3. UI/UX 细节需求

### 3.1 终端交互体验 (TUI Experience)

- **输入稳定性**: 确保在高频输入或 AI 流式输出期间，终端不会丢失用户键入的字符 (即 "Input Race Condition" 修复)。
- **焦点自动切换**: 当进入 HITL 状态时，方向键应自动从 "切换历史记录" 映射为 "菜单选择"。
- **视觉反馈**:
  - 使用 `ink-spinner` 展示 AI 正在处理的状态。
  - 在 HITL 期间隐藏输入框的闪烁光标，以提供明确的模态指示。

### 3.2 渲染稳定性

- **消息持久化**: 历史消息通过 `Static` 组件渲染以节省性能，但带有待处理中断的消息必须保持在 active 区域。
- **防闪烁/复位**: 使用稳定的 React Key (如基于消息时间戳的任务 ID) 渲染消息，防止在流式更新期间列表项复位或闪烁。

## 4. 技术约定与约束

### 4.1 环境配置 (Environment)

- **双层配置文件**:
  1. 项目根目录的 `.env` (最高优先级)。
  2. 用户家目录的 `~/.cubectl/.env` (全局次优先级)。
- **密钥安全**: 必须在初始化时优先验证 API Key 配置，缺失时给予清晰引导。

### 4.2 架构实现

- **前端框架**: 使用 `Ink` (React-based TUI framework)。
- **后端架构**: `LangChain` + `LangGraph`。
- **状态管理**: 严格同步 `inputValueRef` 和 `cursorRef` 以保持 TUI 输入与 React 状态的一致性。

### 4.3 类型安全 (Type Safety)

- **消除 Any**: 全面弃用 `any` 和 `unknown` 类型。
- **官方类型对齐**: 必须使用 LangChain/LangGraph 官方提供的类型，如 `ReactAgent`, `BaseMessage`, `AIMessage`, `ToolMessage` 等。
- **序列化识别**: 消息类型识别不应仅依赖 `instanceof`（在反序列化场景下可能失效），应结合 `_getType()`、`type` 属性以及序列化 ID 路径进行健壮性判断。

## 5. 设计演进与折衷 (Design Evolution & Compromises)

本节记录了项目开发过程中的重大设计变更及由于技术限制所做的折衷。

### 5.1 HITL 菜单位置偏移

- **演进**: 最初计划在每个 Assistant 消息块内直接嵌入 `SelectInput`。
- **折衷**: 由于 Ink 的 `Static` 组件在消息变为非活动状态时会禁用所有交互，且在消息流中间插入交互组件会导致焦点管理极度复杂，最终决定将决策菜单**固定在底部输入区域**。
- **收益**: 确保了交互ের绝对可靠性，并统一了用户的操作预期。

### 5.2 滚动与归档逻辑简化

- **演进**: 曾考虑实现复杂的自动滚动逻辑以处理长对话。
- **折衷**: 鉴于 TUI 环境下对动态高度容器滚动的支持有限，项目采用了**“活动消息 vs. 静态历史”**的二分法。
- **细节**: 只有当前流式输出的消息和带有待处理中断的消息保持在 React 的活动渲染树中，其余消息进入 `Static` 模式以提升性能，避免了复杂的滚动计算。

### 5.3 从 useInput 回退到原生 stdin 监听

- **演进**: 为了标准化组件交互，曾一度全面迁移到 Ink 的 `useInput` 钩子。
- **折衷**: 实践发现 `useInput` 在处理某些终端（尤其是 macOS）的退格键（Backspace, `\x7f`）时存在局限，且在高频输入下容易产生竞态冲突。
- **最终方案**: 回退至使用 **`stdin.on('data')` + 自定义 `key-parser`** 的底层监听方案。
- **收益**: 彻底解决了退格键失灵和输入丢失的问题，同时通过 `isProcessing` 和 `hasInterrupt` 逻辑锁实现了与 AI 流及 HITL 菜单的完美兼容。

### 5.4 消息 Key 的稳定性

- **演进**: 早期使用数组索引作为 React Key。
- **优化**: 为防止消息列表更新（如历史回溯或流式追加）导致的组件重新挂载（Remount），改为使用**基于时间戳的唯一 ID**，确保了 selection 状态在重绘时得以保留。

### 5.5 多轮对话中的消息重复修复

- **演进**: 初步实现的 ID 去重仅在流结束后生效，导致 HITL 期间重复的消息块被错误累加。
- **修复**: 将去重检查点前置到流处理循环的**每一个 Chunk 解析**阶段。
- **序列化兼容**: 针对 LangGraph 回传的序列化历史（`HumanMessage` 等被封装在 `kwargs` 中），增强了 ID 提取和角色识别算法，确保跨 Turn 状态刷新时不会产生 "Unknown Role" 错误。
